01  /  System Architecture

Unified Cross-Platform Framework

To avoid maintainability issues with separate codebases, we developed a Hardware-Aware Profile System. The engine detects the target platform at runtime and swaps core logic modules without rebuilding the scene.

Input Abstraction Layer

A centralized middleware that maps OpenXR motion events and traditional HID inputs to the same gameplay 'Action Registry'.

Scalable Render Profile

Dynamic adjustment of FFR (Fixed Foveated Rendering), post-processing stacks, and texture resolution based on thermal headroom.

Physics Proxy System

Swaps heavy mesh-colliders for simplified primitive sets on mobile chipsets while maintaining identical gameplay collision logic.

02  /  Challenges & Solutions

Engineering Mobile VR Parity

01

Standalone VR CPU Bottlenecks

Problem

High Draw Calls in Dense Horror Environments

Rendering detailed horror environments with complex lighting on the Quest 2's XR2 chipset often exceeded the 100-draw-call budget, causing severe frame drops and motion sickness.

Solution

SRP Batching & Custom Lightprobe Interpolators

We implemented aggressive SRP batching and replaced real-time point lights with a custom lightprobe interpolator. This allowed us to simulate dynamic lighting for 50+ objects in a single draw call.

02

Input Feeling Disparity

Problem

Flat Feeling on Mouse/Keyboard vs VR

The 'tactile' nature of VR combat (manual reloading, physical aiming) felt lost when translated to traditional inputs, making the PC version feel like a generic shooter.

Solution

Simulated Inertia & Micro-Action Mapping

We added simulated weapon inertia to mouse movement and mapped 'Micro-Actions' (like pulling a slide) to specific key combinations. This preserved the 'mechanical' feel of the VR version for PC players.

03  /  Engineering Takeaways

Validation & Results

1

Hardware-Aware Optimization is Mandatory

Designing for the most constrained platform (Quest 2) first, and then scaling up for PC, resulted in a much cleaner architecture than attempting to 'port down' later in development.

2

Unified Codebase for Faster Iteration

By using an abstraction layer for input and rendering, we were able to fix bugs once and deploy to all platforms simultaneously, reducing QA cycles by nearly 35%.

3

Immersion is a Performance Metric

In horror, frame stability is as important as visual fidelity. A locked 72 FPS with slightly lower detail is always superior to a stuttering 90 FPS with high-end assets.
