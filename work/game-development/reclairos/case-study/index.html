01  /  Project Overview

Design Pillars

Reclairos is a 2D story-driven Tower Defense RPG for Android and iOS using Unity URP. The game blends lane-based tactical defense with character-driven RPG progression and an episodic narrative structure — giving players a reason to fight beyond pure survival.

Pillar Description
Story FirstNarrative context drives every combat encounter. Players understand why they are fighting, which deepens strategic investment.
Escort as ObjectiveLevels task the player with safely guiding a unit to the exit — creating dynamic, moving threat responses instead of static base defense.
Class-Based CombatPlayer units are organised into distinct combat roles. Strategic class combinations are essential to progressing through harder encounters.
Mobile-Aware DesignAll systems designed with mobile constraints: touch controls, short session lengths, and stable performance on mid-range hardware.
Scalable ArchitectureNew enemies, story chapters, and unit types can be introduced without rewriting core gameplay systems.
02  /  System Architecture

9-Layer Modular System

Reclairos is built on a modular gameplay architecture where independent systems communicate through well-defined interfaces. This decoupling allows each system to be developed, tested, and extended independently without breaking adjacent functionality.

Layer 1

Player Input

Touch input handler, lane selection, unit drag-and-drop placement, ability trigger buttons.

Layer 2

Character Controller

Unit movement along lanes, escort unit pathfinding, collision and obstacle response.

Layer 3

Combat & Ability System

Damage resolution, ability cooldowns, targeting logic, status effects (stun, slow, burn).

Layer 4

Enemy AI System

Archetype state machines (melee, ranged, disruptor, elite), threat detection, attack sequencing.

Layer 5

Encounter Manager

Wave sequencing, enemy spawning, difficulty scaling, win/loss condition evaluation.

Layer 6

Game State Manager

Session state machine: Lobby → Story → PreBattle → Combat → Results → Progression.

Layer 7

UI Feedback System

Combat HUD, health bars, damage numbers, ability indicators, wave counter, escort progress bar.

Layer 8

Narrative System

Dialogue engine, cutscene sequencer, story state tracker, chapter unlock logic.

Layer 9

Unity Engine (URP)

2D rendering, sprite atlasing, particle VFX, physics, audio, scene management.

Architecture Principles

Modular Systems

The Combat System does not directly reference Enemy AI. Both communicate through a shared DamageEvent / StateChange event bus — preventing tight coupling.

Encounter-Driven Design

A single Encounter Manager reads a data asset describing wave timing, enemy types, and spawn positions — generating the entire encounter at runtime. No hand-scripted scenes.

Scalable by Default

Adding a new enemy type, ability, or story chapter requires creating a new ScriptableObject data asset — not modifying core gameplay code.

03  /  Technical Challenges & Solutions

Four Core Engineering Problems

01

Large Character Roster Scalability

Problem

Unique Animator Controllers Don't Scale

  • Each character class has distinct attack, idle, move, hit-reaction, and death animations
  • Separate Animator controllers per unit create state machine duplication and balloon memory
  • Adding a new unit type required rebuilding an animation graph from scratch
  • Enemies share behavioral patterns but have visually distinct presentations
Solution — Animator Override Controller

One Base State Machine, Per-Unit Clip Overrides

  • Base Animator defines all states and transitions once — shared by every unit
  • AnimatorOverrideController swaps animation clips per unit type at instantiation time
  • New unit types require only a new clip set and override asset — no state machine authoring
  • Result: 40% animator memory reduction — one base controller in memory, override assets are lightweight clip references
02

Combat Readability on Mobile Screens

Problem

Visual Chaos at Scale

  • Mobile screens are 5–6 inches — multiple overlapping characters, projectiles, and effects compete for attention
  • Players must distinguish allies from enemies, identify enemy archetype at a glance, and track the escort unit at all times
  • VFX density during large encounters can obscure unit positions and health states
  • Damage number spam becomes unreadable during intense waves
Solution — Visual Hierarchy System

Readability as a First-Class Engineering Concern

  • Escort unit always rendered on a dedicated top sorting layer — never occluded by other units or effects
  • Ally and enemy units use distinct silhouette shapes and colour-coded UI rings for immediate archetype identification
  • VFX particle budgets capped per encounter by the Encounter Manager — effects are pooled and reused, never spawned unlimitedly
  • Damage numbers use a priority queue: only the most recent N values are displayed simultaneously
03

Mobile Performance Constraints

Problem

18 FPS on Snapdragon 665 Hardware

  • Each active unit requires an Animator, Collider, and AI tick every frame
  • Particle effects from abilities multiply GPU overdraw significantly
  • Unity Editor profiling underestimated real-device performance by a wide margin
  • GC allocation patterns caused frame hitches during encounter-start instantiation
Solution — Multi-Layer Optimisation

Stable 60 FPS on Snapdragon 665 Achieved

  • Sprite atlasing (Unity Sprite Atlas) — all unit sprites packed per chapter to minimise draw calls from 120+ to under 30
  • Object pooling for all units, projectiles, VFX, and damage numbers — zero runtime Instantiate() calls during combat
  • AI tick staggering — enemy state machines update on a distributed tick schedule, not every frame
  • Animator culling set to 'Cull Completely' for off-screen units — zero evaluation cost for unseen characters
  • Particle systems converted to URP Particles Unlit shader — removes unnecessary lighting calculations
04

System Scalability Under Growing Content

Problem

Hard-Coded Logic Blocks Scale

  • Hard-coded enemy behaviour logic becomes a maintenance burden with 10+ enemy types
  • Encounter scripts that reference specific unit types cannot be reused across chapters
  • Ability systems embedding damage values in code block rapid balancing iteration
  • Every balance change required a code modification, recompile, and QA pass
Solution — ScriptableObject Architecture

Gameplay Configuration Separated from Code

  • EnemyArchetypeSO: movement speed, attack range, damage, health, behavior state weights, visual asset reference
  • WaveDefinitionSO: enemy archetype list, spawn intervals, spawn positions, difficulty modifier
  • AbilitySO: cooldown, targeting rule, effect type, damage/heal values, status effect applied
  • New enemy types, abilities, and encounters created entirely in Unity Editor — no C# files touched
  • Balance tuning reduced from hours (recompile + QA) to minutes (editor SO edit)

05  /  Engineering Takeaways

Seven Lessons Validated in Production

1

Modular Architecture Enables Long-Term Scalability

Adding the third story chapter required zero changes to the Combat, AI, or Encounter Manager systems — only new ScriptableObject data assets. This validated the modular approach and significantly reduced chapter delivery time compared to a monolithic codebase.

2

Data-Driven Design Is a Force Multiplier

ScriptableObjects allowed non-engineer team members to author and balance enemies, abilities, and encounters directly. Balance iteration cycles that would have required a recompile and QA pass became editor operations measurable in minutes rather than hours.

3

Escort Mechanics Demand Different Threat Modeling

Traditional tower defense threat modeling focuses on static base defense. Escort design requires the threat model to account for a moving, vulnerable variable. Every system — unit placement, enemy targeting, wave pacing — had to be re-evaluated around the escort's position. The result: fundamentally more engaging gameplay.

4

Combat Readability Is a First-Class Engineering Concern

Readability was initially treated as a UI/art problem. Midway through development it became clear it was an engineering problem: particle budgets, sorting layers, damage number queuing, and VFX pooling all directly affected whether the player could read the battlefield under pressure. These systems needed explicit engineering ownership, not just art direction.

5

Mobile Performance Must Be Validated on Real Hardware Early

Unity Editor profiling significantly underestimated the performance cost of the encounter system on mid-range Android devices. Frame budgets appearing comfortable in editor were failing target frame rate on Snapdragon 665 hardware. A weekly on-device profiling ritual from the first playable build prevented performance debt from accumulating silently.

6

The Animation Override Pattern Is Underutilised in Mobile RPGs

AnimatorOverrideController is one of Unity's most underused tools for mobile RPGs with large character rosters. The pattern — one base state machine, per-unit clip overrides — delivered 40% memory savings and dramatically reduced the engineering cost of onboarding a new unit type. It should be the default approach for any mobile title with more than five character archetypes.

7

Story Context Amplifies Tactical Investment

The most consistently positive playtest feedback was not about mechanics — it was about caring which characters survived. Players who had engaged with the narrative sequences were measurably more engaged with combat outcomes. Story integration is not merely a content layer; it is a motivation layer that makes tactical gameplay feel consequential rather than abstract.
